# 🤖 多智能体协调模式入门

> **目标读者：** 对 AI 智能体只有初步了解的同学
> **读完本文你能学到：** 多个 AI 智能体之间是怎么"打配合"的，以及什么场景该用什么模式

---

## 📖 先说说什么是"多智能体"

你有没有这种感觉——

用一个 AI 工具的时候，它啥都能干，但一让它同时做**很多件有关联的事**，就开始出错或者乱了套？

这就对了。一个 AI 再强，也有极限。

**多智能体（Multi-agent）** 的思路很简单：
> 不让一个 AI 单打独斗，而是找来**多个 AI**，各有分工，互相配合。

就像一个公司里有不同岗位的员工，各干各的，最后汇总成完整的工作成果。

---

## 🎯 五个基本配合模式

下面介绍五种常见的配合方式，从简单到复杂。

---

## 模式一：🤝 生成器-验证器

**一句话理解：** 一个写，一个检查，写完检查，检查不过打回重写。

### 工作流程图

```
┌────────────┐          ┌────────────┐
│            │  ──产出─▶ │            │
│   生成器    │          │   验证器    │
│   (写手)   │ ◀─反馈──  │   (审核)    │
└────────────┘          └────────────┘
        │                      │
        │◀────── 循环重试 ──────│
        │                      │
        └──────── ✅ 通过 ───────┘
```

### 通俗解释

- **生成器**：负责干活、产出结果
- **验证器**：负责把关、挑毛病
- 验证器说"不行"，生成器就重来，直到验证器说"可以"为止

### 现实类比

就像**考试和阅卷**：
- 学生做题 → 生成器
- 老师批卷 → 验证器
- 不及格？打回去重做！及格了才通过

### 适合哪些场景

| 场景       | 说明                   |
|----------|----------------------|
| 📧 自动写邮件 | 一个写，一个检查有没有错别字、语气对不对 |
| 💻 写代码   | 一个写代码，一个写测试用例并运行     |
| ✅ 合规审查   | 检查合同/文档是否符合规定        |
| 📊 数据核对  | 核对报表数据是否准确           |

### 优点 | 缺点

| 优点 ✅      | 缺点 ❌                |
|-----------|---------------------|
| 简单直接，效果好  | 验证标准不好定，定不清就成了"走过场" |
| 质量有保障     | 如果验证和生成一样难，验证器就形同虚设 |
| 反复打磨，直到过关 | 可能无限循环，需要设最大次数      |

## 模式二：👨‍💼 编排器-子智能体

**一句话理解：** 一个老板 + 若干员工。老板分派任务，员工干活，老板汇总。

### 工作流程图

```
                 ┌──────────────┐
                 │    编排器     │
                 │   （主脑）    │
                 └──────┬───────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
   ┌─────────┐     ┌─────────┐     ┌─────────┐
   │子智能体A │     │子智能体B │     │子智能体C  │
   │  (员工) │     │  (员工)  │     │  (员工)  │
   └────┬────┘     └────┬────┘     └────┬────┘
        └───────────────┼───────────────┘
                        │
                 ┌──────▼──────┐
                 │    汇总结果    │
                 └──────────────┘
```

### 通俗解释

- **编排器（Orchestrator）**：团队领导，负责规划、分工、汇总
- **子智能体（Subagent）**：员工，各自从编排器接活，干完汇报
- 编排器是唯一的大脑，员工只管自己的那一块

### 现实类比

就像**建筑项目**：
- 项目经理（编排器）：拿到图纸，规划要挖地基，建墙、装修，分配给不同班组
- 各班组（子智能体）：各干各的，干完汇报
- 项目经理汇总成完整项目

### 适合哪些场景

| 场景      | 说明                          |
|---------|-----------------------------|
| 🔍 代码审查 | 安全检查、风格检查、覆盖率检查——每项独立，可同时进行 |
| 📝 综合报告 | 收集各地区、各部门数据，汇总成一份报告         |
| 📦 项目管理 | 分解任务、分派、追踪、汇总               |

### 优点 | 缺点

| 优点 ✅        | 缺点 ❌                             |
|-------------|----------------------------------|
| 结构清晰，容易理解   | 编排器成了"信息高速公路"——员工之间的信息要经过它，容易丢细节 |
| 适用范围广       | 员工之间难以直接沟通，必须绕道编排器               |
| 适合任务分解明确的场景 | 默认顺序执行，没有并行加速                    |

## 模式三：👥 智能体团队

**一句话理解：** 不是临时工，是正式员工！团队成员常驻，持续积累经验。

### 工作流程图

```
        ┌──────────────────┐
        │      协调者       │
        │   (Coordinator)  │
        └────────┬─────────┘
                 │
          投放任务到队列
                 ▼
        ┌──────────────────┐
        │    共享任务队列    │
        │   (Task Queue)   │
        └────────┬─────────┘
       工人取     工人取     工人取
        ▼         ▼         ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ 常驻    │ │ 常驻     │ │ 常驻     │
   │  Worker │ │  Worker │ │  Worker │
   └─────────┘ └─────────┘ └─────────┘
```

### 通俗解释

- **协调者（Coordinator）**：派活的人，不直接干活
- **Worker（员工）**：常驻的，一直活着，不断接任务，越干越熟练
- 关键区别：员工不是"用完就消失"，而是**积累经验和上下文，越用越强**

### 现实类比

就像**软件开发团队**：
- Team Lead（协调者）：分配任务给各个开发
- 各个开发（Worker）：不是干完就走，而是长期驻扎项目，不断积累对代码库的理解，越熟悉越高效

> 对比一下：**模式二**的员工用完就bye-bye，下次来新人从头学起。
> **模式三**的员工一直干，越干越懂业务。

### 适合哪些场景

| 场景            | 说明                    |
|---------------|-----------------------|
| 🏗️ 大型代码库迁移   | 每个服务分配给一个 worker，独立迁移 |
| 📊 批量数据分析     | 多个数据集同时分析，各管各的        |
| 🔄 长期运营的自动化流程 | 任务不断来，worker 持续在线     |

### 优点 | 缺点

| 优点 ✅               | 缺点 ❌                 |
|--------------------|----------------------|
| Worker 越用越强，效率越来越高 | 要求任务之间**真正独立**，否则会冲突 |
| 并行处理，效率高           | 员工之间无法直接沟通中间发现       |
| 适合长时任务             | 完成时间不一致，有人2分钟，有人20分钟 |

## 模式四：📬 消息总线

**一句话理解：** 不再指定谁找谁，而是"大家订阅感兴趣的话题，来了消息自动分发"。

### 工作流程图

```
  ┌──────────┐          ┌──────────┐          ┌──────────┐
  │ 智能体A   │          │ 智能体B   │          │ 智能体C   │
  └───┬──────┘          └───┬──────┘          └───┬──────┘
      │ 订阅:网络            │ 订阅:身份             │ 订阅:响应
      │◀───────────────────┼─────────────────────▶│
┌─────┴────────────────────────────────────────────────▼────────┐
│                     📮 消息总线                                │
│                  (Message Bus / Router)                       │
│                                                               │
│   网络告警 ─────────────────▶ [网络分析智能体]                    │
│   身份告警 ─────────────────▶ [身份分析智能体]                    │
│   上下文请求 ───────────────▶ [上下文收集智能体]                   │
└────────────────────────────────────────────────────────────────┘
```

### 通俗解释

- **发布（Publish）**：智能体把消息扔到总线上
- **订阅（Subscribe）**：智能体声明"我对XX类型消息感兴趣"
- **路由器（Router）**：决定消息发给谁
- 新成员加入？直接订阅即可，不用改动老成员

### 现实类比

就像**公司内部论坛或钉钉群**：
- 有人发了一条"网络故障"帖子
- 负责网络的团队成员自动收到通知
- 其他人不受影响
- 来了新同事？拉进群就行，不用改群结构

### 适合哪些场景

| 场景           | 说明                     |
|--------------|------------------------|
| 🚨 安全运营      | 告警进来 → 自动分诊 → 路由给对应调查员 |
| 📡 物联网数据处理   | 传感器数据源源不断，各处理程序各取所需    |
| 📈 事件驱动的业务流程 | 事件触发下一步，不需要固定顺序        |

### 优点 | 缺点

| 优点 ✅        | 缺点 ❌                     |
|-------------|--------------------------|
| 新成员加入不用重新布线 | 追踪困难——一个事件触发5个智能体，出问题难排查 |
| 天然解耦，易扩展    | 路由器出错就静默失败——不崩溃但也没干活     |
| 适合事件驱动的场景   | 基于 LLM 的路由有它自己的问题        |

## 模式五：🗄️ 共享状态

**一句话理解：** 没有老板！大家读写同一本"工作手册"，各自看到最新进展。

### 工作流程图

```
    ┌─────────────────────────────────────────────────────┐
    │              🗄️ 共享数据库 / 共享文档                  │
    │  ┌───────────────────────────────────────────────┐  │
    │  │  📝 问题：分析竞品情况                           │  │
    │  │  👤 智能体A：学术文献 ────────────── [已完成]     │ │
    │  │  👤 智能体B：行业报告 ────────────── [进行中]     │ │
    │  │  👤 智能体C：专利查询 ────────────── [等待中]     │ │
    │  │  👤 智能体D：新闻监控 ────────────── [进行中]     │ │
    │  └───────────────────────────────────────────────┘  │
    └─────────────────────────────────────────────────────┘
         ▲            ▲            ▲            ▲
         │ 写         │ 写         │ 读/写      │ 读
    ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
    │ 智能体A  │  │ 智能体B  │  │ 智能体C  │  │ 智能体D  │
    │  (学术)  │  │  (行业) │  │  (专利)  │  │  (新闻)  │
    └─────────┘  └─────────┘  └─────────┘  └─────────┘
```

### 通俗解释

- **没有中央协调者**
- 智能体们共用一个"数据库"或"文档"
- 每个智能体想：**"写"发现进去 → 其他人"读"到 → 再写自己的发现**
- 直到某个终止条件满足才停下（时间到了、或大家一致认为"差不多了"）

### 现实类比

就像**科研团队协作写论文**：
- 共享一个 Google Doc
- 小王写文献综述
- 小李写行业分析
- 小张写专利分析
- 每个人随时能看到别人写了啥，随时补充自己的部分

### 适合哪些场景

| 场景        | 说明                 |
|-----------|--------------------|
| 🔬 协作研究   | 多个智能体研究不同方面，互相参考发现 |
| 📚 深度调研   | 市场调研、技术调研、用户调研同步进行 |
| 🧠 复杂问题分析 | 问题太大，一个智能体装不下，分工研究 |

### 优点 | 缺点

| 优点 ✅     | 缺点 ❌                                  |
|----------|---------------------------------------|
| 完全没有单点故障 | 没有显式协调，可能重复劳动或结论矛盾                    |
| 没有信息瓶颈   | **最怕"反应性循环"**——A写→B读后写→A又读→无限循环烧token |
| 实时看到彼此进展 | 必须设定终止条件（时间/轮数/共识）                    |

## 📊 一图总结

```
模式                  信息谁来管？          适用场景                复杂度
──────────────────────────────────────────────────────────────────────────
生成器-验证器        一对一循环           质量关键、标答清晰       ⭐
编排器-子智能体      中央集权             任务分解清晰            ⭐⭐
智能体团队           放权给常驻员工       并行独立长时任务         ⭐⭐⭐
消息总线             路由器分发           事件驱动、生态增长      ⭐⭐⭐
共享状态             去中心化数据库       协作研究、实时共享       ⭐⭐⭐⭐
```

## 💡 什么时候选什么？

### 按场景选

| 你需要……               | 推荐用      |
|---------------------|----------|
| 质量很关键，能明确说"什么算过关"   | 生成器-验证器  |
| 任务能拆成几个独立小任务        | 编排器-子智能体 |
| 任务独立但量大，需要并行处理      | 智能体团队    |
| 工作流程是"事件驱动"，来了啥就处理啥 | 消息总线     |
| 多个智能体要互相参考、实时协作     | 共享状态     |

### 按经验选

```
新手建议从"编排器-子智能体"开始 ✅

搞清楚了 → 按需演进到其他模式

不要一上来就上最复杂的模式！
```

## 🏠 实际可以怎么用？

**举几个例子：**

**1. 日报处理（适合"编排器-子智能体"）**
- 主智能体：读取邮件，分类哪些是晟哥的待办
- 子智能体A：分析邮件内容，提取任务
- 子智能体B：查相关文档、表格
- 主智能体汇总成待办清单

**2. 竞品调研（适合"共享状态"）**
- 智能体A：查网络上的竞品分析
- 智能体B：查行业报告
- 智能体C：查用户评价
- 各自写进共享文档，互相补充

**3. 数据质量检查（适合"生成器-验证器"）**
- 生成器：清洗数据、输出报表
- 验证器：核对数据口径、逻辑是否正确
- 不通过就打回重洗

## ⚠️ 避坑指南

| 坑             | 说明               | 解决方案          |
|---------------|------------------|---------------|
| 选了太复杂的模式      | 本来一个智能体能搞定，非要上5个 | **从小开始**，不行再加 |
| 验证器标准定不清      | 验证器形同虚设          | 把"什么算过关"写清楚   |
| 忘了设终止条件       | 智能体无限循环烧token    | 一定要设最大轮数/时间   |
| 任务不够独立却选了团队模式 | worker 之间打架      | 任务真独立才用此模式    |
| 消息总线路由器出错     | 静默失败，消息丢了        | 做好监控和日志       |

## 💡 核心本质：上下文谁来管、管什么

很多人以为多智能体的核心是"分工"，但真正核心是**上下文（Context）谁来管、管什么**。

每个 AI 智能体能处理的上下文是有限的（上下文窗口有上限）。多智能体的本质，是**在多个智能体之间切分上下文，让每个智能体只关心自己那块，减少每个智能体的上下文负担**。

五种模式的本质区别，就是切分方式不同：

| 模式           | 上下文谁来管      | 切分方式                        |
|--------------|-------------|-----------------------------|
| **生成器-验证器**  | 不切分，循环迭代    | 同一个上下文反复打磨，通过反馈缩小范围         |
| **编排器-子智能体** | 编排器集中管      | 上下文集中在编排器，子智能体各自独立但不共享      |
| **智能体团队**    | worker 各持一块 | 按任务切分，每个 worker 持自己的上下文，不交叉 |
| **消息总线**     | 按事件类型分发     | 上下文按"消息类型"路由到不同的订阅者         |
| **共享状态**     | 去中心化        | 上下文写入共享存储，每个智能体看到的是同一个世界    |

所以选模式的问题，本质上是：**你的上下文该怎么切分？**

## 🔄 如何从简单演进到复杂？

原文的核心建议是：**从最简单能解决问题的模式开始，观察它在哪里吃力，再演进。**

演进的核心判断依据是：**当前模式卡在哪里？**

### 演进路径图

```
编排器-子智能体（起步首选）
    │
    ├── 觉得编排器成了瓶颈 ──────▶ 智能体团队（放权给 worker）
    │                                或
    │                              共享状态（去掉中央协调）
    │
    ├── 任务之间不独立，                 
    │   worker 打架 ───────────────▶ 消息总线（事件驱动解耦）
    │
    └── 输出质量总是不达标 ───────▶ 生成器-验证器（加验证循环）
```

### 实战演进案例

**场景：** 用多智能体做代码审查

1. **起步：** 用编排器-子智能体，主智能体分派安全检查、风格检查、覆盖率检查给三个子智能体
2. **发现问题：** 子智能体A 发现的安全漏洞，需要让子智能体B 知道，但信息绕编排器转发后失真了
3. **演进：** 如果任务相对独立 → 升级到智能体团队；如果需要实时协作 → 升级到共享状态

### 演进检查清单

| 问题                | 意味着该考虑……      |
|-------------------|---------------|
| 编排器成为信息瓶颈         | 智能体团队 / 共享状态  |
| 任务之间不独立，worker 冲突 | 消息总线（解耦）      |
| 输出质量总是不过关         | 生成器-验证器       |
| 需要横向往更多场景扩展       | 消息总线（容易加新智能体） |
| 智能体之间需要实时看到彼此进展   | 共享状态          |

**记住：** 演进不是升级，不是越复杂越好。演进是解决当前模式的实际问题。

---

## 📝 小结

1. **多智能体不是银弹**——能用一个搞定的，不要硬上多个
2. **从最简单的开始**，观察哪里不行，再演进
3. 选模式看三点：**任务能否分解、独立性需求、要不要实时协作**
4. **必须有终止条件**，不然 token 烧到你心疼
5. **核心是上下文的切分**——不同模式对应不同的切分方式
